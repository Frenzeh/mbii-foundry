# Definitions Contribution Guide

**Fixing typos, clarifying prose, or replacing AI-generated nonsense in MBII Foundry's enum documentation.** This is the single highest-leverage contribution anyone can make — it doesn't require Go, it doesn't require compiling anything, and every definition you polish improves the app's in-panel help for every user forever.

---

## What "definitions" are

Every game enum the Foundry knows about (a force power, a weapon, a class, an attribute, a class flag) has:

1. A JSON row in `data/<category>.json` — the machine-readable metadata (id, level count, default values).
2. A markdown file in `definitions/<category>/<ID>.md` — the human-readable prose (what it does, per-level effects, tips).

You're editing the **markdown**. That's where the words live. The JSON is maintained automatically by a maintainer running a regen tool against the game's C source.

Categories you'll see:

| Folder | Contents | Typical file |
|---|---|---|
| `definitions/attributes/` | `MB_ATT_*` character attributes | `MB_ATT_PUSH.md` |
| `definitions/weapons/` | `WP_*` weapons | `WP_BLASTER.md` |
| `definitions/mb_classes/` | `MB_CLASS_*` player classes | `MB_CLASS_JEDI.md` |
| `definitions/force_powers/` | Force power metadata | `MB_ATT_FP_HEAL.md` |
| `definitions/saber_styles/` | Saber stance docs | `SS_STRONG.md` |
| `definitions/class_flags/` | `CFL_*` class modifiers | `CFL_BPFREEJUMPS.md` |
| `definitions/holdables/` | `HI_*` inventory items | `HI_BACTA.md` |
| `definitions/glossary/` | Field-name explanations (not enums) | `uishader.md` |

## Finding what to work on

**Stubs** — files that were auto-generated and only contain placeholder text — are the top priority. They look like this:

```markdown
# MB_ATT_BACTA

`MB_ATT_BACTA`

*Stub — a human needs to document this.*

Source: `game/bg_public.h` (auto-generated by `fa_generate_definitions`).
```

To list every stub in one shot:

```bash
cd mbii-foundry
grep -rlF "*Stub — a human needs to document this.*" definitions/
```

You'll get a long list. Pick one you know well (from playing the game) — that's your first contribution.

**Real definitions that may be wrong** — harder to spot. AI-generated or hastily written prose can look confident but contain hallucinations. The classic tells:
- Numbers that don't match what the game actually does (damage, cooldown, cost).
- Level descriptions that swap effects between levels.
- References to attributes/weapons that don't exist under that name.
- Overly flowery language that doesn't say anything concrete.

When in doubt, **test the thing in-game** before rewriting its description.

## What "good" looks like now

**Update (2026-04-22):** the bar raised. Prose-only descriptions like "Legendary Mandalorian iron plating" read as lore, but what a balance designer actually needs are **numbers**: damage multipliers per level, FP costs, cooldowns, what's NOT affected, source-code references. Flavor prose without stats is no longer considered a complete definition.

The reference template is [`definitions/attributes/MB_ATT_BESKAR.md`](../definitions/attributes/MB_ATT_BESKAR.md). Every new or revised definition should follow its shape where applicable:

1. `# <emoji> <Display Name>` — one-line title
2. `` `<ENUM_ID>` `` — the exact enum as it appears in source
3. One plain-English line summarizing what it does (for new players)
4. Any class-specific / unlock requirements as bold metadata
5. **`### Mechanics` or a stats table** — the mandatory part. Actual multipliers / costs / thresholds / rates. One row per level if it's a leveled attribute.
6. `### What isn't affected` or exclusions — as important as the positive effects. Balance designers need to know the edges.
7. `### Related` — cross-links to interacting attributes (`MB_ATT_ARMOUR`, `CFL_*`, etc.)
8. `### Source references` — file + line ranges in the MBII C source so future reviewers can verify. Even a single `game/bg_weapons.h:1065-1084` is valuable.

Run `tools/audit-definitions.py` to see the current quality rating for every entry; the report is checked in at `docs/DEFINITION_QUALITY_AUDIT.md`.

## Anatomy of a good definition — older format

Compare a stub:

```markdown
# MB_ATT_PUSH

`MB_ATT_PUSH`

*Stub — a human needs to document this.*
```

…to an earlier "prose-polished" entry like `MB_ATT_AMMO.md`:

```markdown
# 🔋 Ammo

`MB_ATT_AMMO`

Increases the starting ammunition pool for all equipped weapons.

> **Default:** Varies by weapon (usually enough for ~2-3 reloads).

### 📊 Levels

*   **Level 1**: **Extra Clips**. Good for prolonged firefights.
*   **Level 2**: **Heavy Loadout**. Significant ammo reserve.
*   **Level 3**: **Maximum Capacity**. Almost never run out of ammo.

### 💡 Tips
*   Critical for rapid-fire weapons like the **Clone Rifle** or **T-21**.
*   **SBDs** use Battery charge instead of Ammo, but this attribute increases their Battery capacity (`MB_ATT_RECHARGE` affects the regen rate).
```

The template that works:

```markdown
# [emoji] Short Display Name

`ENUM_ID_AS_IN_CODE`

One-sentence plain-English summary. What does this do, in a sentence a new player understands?

> **Default:** Anything a reader ought to know before reading further — base cost, default value, what happens if nobody picks this.

### 📊 Levels   (only if the attribute has levels)

*   **Level 1**: **Short name**. What it does at this level. Include concrete numbers if you know them.
*   **Level 2**: **Short name**. …
*   **Level 3**: **Short name**. …

### 💡 Tips     (optional but encouraged)
*   When should someone pick this? When is it a trap? What synergizes?
*   Any common confusions to clear up (e.g. "this is different from MB_ATT_OTHER which looks similar").
```

Emoji headers match the existing style; keep using them. Pick something evocative:
- Force powers: ⚡ ✨ 💫 🌊 🛡️ 🩹
- Weapons: 🔫 ⚔️ 💥 🎯
- Defensive attrs: 🛡️ 🦺
- Movement: 💨 🧗 🕊️
- Economy / utility: 🔋 💊 📡

## Verifying your claims before writing

**Ordered by authority:**

1. **In-game testing.** The most reliable. Spin up a dev server (`/devmap mb2_dotf`), pick a class that has the attribute, feel what it does. Your memory of how the game plays beats any source code citation for prose about *how it feels*.

2. **MBII Wiki** — [moviebattles.fandom.com](https://moviebattles.fandom.com/wiki/Moviebattles_Wikia). Player-facing community docs. Good for class-level context, less good for per-attribute minutiae.

3. **MBII source code**, if you have access. Look in `game/bg_public.h` for the enum definition, `game/bg_classes.h` for how classes use it, `game/bg_misc.c` for item/weapon tables, `game/g_active.c` or `g_combat.c` for combat behaviour. The Foundry repo *doesn't* include source — only maintainers with an MBII-dev checkout need to dig here.

4. **The Holocron MCP** (advanced). If you're using an AI assistant with Holocron access, you can query `ask_holocron("What does MB_ATT_PUSH do?")` and it'll return source-grounded answers. Not required — wiki + in-game testing cover 95% of cases.

**Ordered by unreliability:**

- Your AI coding assistant *without* source context. It will make things up with perfect confidence. Always cross-check.
- Old forum threads. MBII balance changes often; a 2019 thread may describe mechanics that no longer exist.
- "I'm pretty sure this is how it works." Test it.

## Common AI mistakes to watch for

Past AI-written stubs have shown recurring errors. If you're reviewing or editing, scan for these:

1. **Level swaps.** Level 1 describes what Level 3 does, and vice versa. Symptoms: "Level 1: Master. Huge knockback." Real: Level 1 is Basic, Level 3 is Master.

2. **Wrong costs.** FP costs quoted as random-looking numbers that don't match the game. Either cite a real number (from testing) or drop the number entirely.

3. **Attribute identity confusion.** `MB_ATT_PUSH` vs `MB_ATT_FORCE_PUSH` vs `EAS_FP_PUSH`. These look similar but aren't the same — the Foundry treats `MB_ATT_*` as the grant (you have this ability) and `EAS_*` as the specific activation alias. Don't merge them.

4. **Made-up synergies.** "Combines well with `MB_ATT_NONEXISTENT`." Delete if you can't verify the referenced enum is real.

5. **Marketing-speak.** "Powerful, versatile, a must-have for any loadout!" — says nothing. Replace with a concrete use case: "Best on Jedi builds with low FP pool who need cheap Push access."

6. **Game mechanic drift.** An AI trained on old MBII forum data may describe pre-R21 mechanics. Cross-check against recent changelog if the description feels off.

## How to submit a fix

### Option A: GitHub web editor (easiest, no git needed)

1. Navigate to the file on GitHub (e.g. `definitions/attributes/MB_ATT_BACTA.md`).
2. Click the **pencil icon** (top-right of file view).
3. Edit the markdown directly.
4. Scroll down, write a short commit message (`Document MB_ATT_BACTA — heal-over-time item`), choose **Create a new branch**, click **Propose changes**.
5. You'll land on a **Create pull request** page. Give it a title, describe your change briefly, submit.

That's it. A maintainer will review and merge.

### Option B: Local edit (if you're already cloned)

```bash
cd mbii-foundry
git checkout -b definitions/mb_att_bacta
# edit the file in your editor of choice
git add definitions/attributes/MB_ATT_BACTA.md
git commit -m "Document MB_ATT_BACTA — heal-over-time item"
git push origin definitions/mb_att_bacta
```

Then open a PR from the GitHub UI.

### Option C: Bulk edits (5+ files at once)

Same as Option B, but one branch, one PR. Keep the changes focused on a single category (e.g. "polish all `definitions/weapons/WP_*_NADE.md` files") so the review is manageable.

## Style conventions

- **Second person.** "You'll want to pick this when…" not "One would want to pick this when…"
- **Present tense.** "Heals 25 HP over 5 seconds" not "Will heal…"
- **Concrete numbers** if you have them; omit if you don't. Wrong numbers are worse than no numbers.
- **Keep lines short.** Readability first; don't wrap prose into 150-column walls.
- **Emoji in headers is fine**, even encouraged. Emoji in body paragraphs is not.
- **No pronouns for the player's gender/style.** "The Jedi character" not "he" or "she".
- **MBII terminology:** *Full Authentic* (not "FA" in prose, but `MB_ATT_` is fine inline), *Siege*, *Open*, *Duel*, *FP* (Force Power), *BP* (Block Points), *AP* (Armor Points).

## Edge cases and judgement calls

- **What if the enum doesn't actually exist in-game?** File flagged as `"orphan": true` in JSON (e.g. `MB_ATT_QUICKTOSS`). Two options:
  - Rename: if you know the enum was renamed to something live, move the prose to the new file and delete the orphan.
  - Annotate: add a note at the top of the `.md` like `*Note: this attribute may have been removed in a recent build; behavior described here is from R21.9.*` — then leave it. A maintainer will decide whether to delete.

- **What if the levels are wrong?** Fix them. In-game testing beats anything written before.

- **What if the attribute does something I don't fully understand?** Write what you *do* know, mark what you don't with `*(behavior under investigation)*`, and move on. Half-right is better than nothing; the next contributor can finish.

- **What if there's an emoji I really hate?** Swap it. They're taste-driven; consistency within a category is nice but not load-bearing.

These markdown files feed:
